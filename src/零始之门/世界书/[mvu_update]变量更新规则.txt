【变量主动写入总原则｜分层版】
1. 输出协议必须成立：只要本轮存在变量更新，就必须按既定 `<UpdateVariable> + <Analysis> + <JSONPatch>` 协议输出，确保变量管理器能读到。
2. 变量更新分为三层，优先级固定如下：
   - 开局建档层：仅用于“同步角色后进入正式剧情的首轮”，负责把基础骨架补齐，避免衣着、背包、地点、角色状态长期空白。
   - 事件驱动层：正式剧情开始后，先判断哪些模块被本轮事件触发，再只更新这些模块。
   - 每轮必写层：少量必须保持活着的核心字段每轮都要检查并在需要时更新。
3. 每轮不要再逐项硬查所有面板。必须先判断本轮触发模块，再更新对应字段；未触发模块默认沿用旧值，不为了“整齐”机械重写。
4. 变量更新不是“有空再写”，而是回复生成流程的一部分。只要某个已触发模块已经形成稳定判断，就必须在本轮写入，不得拖到下一轮。
5. 开局首轮尤其必须主动：
   - 若已完成角色同步并进入正式剧情，则优先写入 1. 世界信息.时间 / 场所 / 天气 / 遭遇事件 / 当前场景标签 / 备注
   - 优先写入 2. 通用状态栏.角色A / 角色B 的神态、动作、希望、内心、衣着、伤势、数值
   - 优先写入 3. 通用状态栏.地图 的 current 节点，并为地点选择具体 emoji
   - 优先写入 4. 背包栏 的具体内容，不得长期空白；至少应依据剧情、角色设定、阵营与场景补入 1~4 个具体物品
   - 优先写入 5. 关系面板.当前关系事件 与必要的好感变化
   - 优先写入 6. NSFW状态栏.总结骚话，哪怕当前并未进入 NSFW
6. 开局默认值采用“稳定生成”思路：
   - 同一角色 + 同一路线 / 同一开局场景下，衣着、基础随身物、常备背包内容优先保持稳定一致
   - 后续只在剧情触发时变化，不得无故漂移或每轮随机改写
7. 自定义开局优先级高于默认模板：
   - 若玩家自行发送自定义开局问候语、行动或场景设定，则一律以玩家输入为准
   - 不得强行回滚到固定问候语、固定地点、固定天气或固定背包
   - 固定问候语只作为可选模板，不是强制剧情起点
8. 背包必须写“具体物名”，不要为了省事一律抽象化：
   - 可以写 巧克力棒、苏打饼干、半瓶咖啡、旧门禁卡、备用弹匣、沾湿绷带、褪色便签 等
   - 只有当剧情确实无法细分时，才使用“压缩口粮”“医疗物资”这类总类名称
   - 必须得有才能使用，手持也占背包位置
9. 地图 emoji 必须具体、贴合地点，不要长期只写 📍。emoji 必须和剧情以及地点挂钩，以下只是参考：
   - 军营/哨所/基地可用 🪖
   - 病房/医院/医务区可用 🩺
   - 仓库/补给站可用 📦
   - 街区/商区可用 🏙️
   - 码头/港口可用 ⚓
   - 检查点/关卡可用 🚧
   - 安全屋/宿舍/房间可用 🛏️
   - 古董店/商店可用 🏪
   - 地铁/隧道/地下可用 🚇
   - 森林/荒野/山地可用 🌲
   - 温泉/浴场可用 ♨️
   - 训练场/竞技区可用 🏟️
   - 临时营地可用 ⛺
10. `总结骚话` 属于每轮必写层：
   - 每轮都必须换新句
   - 非 NSFW 时也要输出，可退化为角色吐槽 / 场景一句话 / 关系余波一句话
   - 禁止复用上一轮原句，禁止固定模板套皮
11. 变量写入必须服务剧情感：内容要具体、贴场景、贴人物，而不是机械填表。
12. 兼容性规则：
   - 对于 计划列表、背包栏.武器详情、背包栏.情报道具详情、通用状态栏.地图、危机面板、NPC档案、怪物档案、关系面板.角色关系、关系面板.盟友、关系面板.敌对、战斗面板.我方、战斗面板.敌方 这类对象/record 字段，新增条目时不要使用 insert。
   - 必须读取当前父对象后，用 replace 一次性写回“合并后的完整对象”。
   - 只有修改现有数值时才优先使用 delta；新增对象项统一使用 replace 父对象。

【开局首轮最低写入清单】
- 世界信息.时间
- 世界信息.场所
- 世界信息.天气
- 世界信息.当前场景标签
- 世界信息.遭遇事件（无事件也写无）
- 通用状态栏.角色A.神态 / 动作 / 希望 / 内心 / 衣着
- 通用状态栏.角色B.神态 / 动作 / 希望 / 内心 / 衣着 （若角色B在当前叙事中成立）
- 通用状态栏.地图.current
- 背包栏中添加至少一个子栏位的具体物品
- 背包栏.本轮变化
- 关系面板.当前关系事件
- NSFW状态栏.总结骚话

【正式剧情轮默认规则】
- 每轮先判断哪些模块被触发，再只更新这些模块。
- 世界信息、通用状态栏.角色A / 角色B 当前状态、关系面板.当前关系事件、判定状态.最近结果、NSFW状态栏.总结骚话 属于每轮必写核心。
- 神态 / 动作 / 内心 按“当前主变化”更新；允许只更新其中 1~2 个，不要求每轮整组重写。
- 希望只在当前直接目标明显变化时更新，不随每句台词抖动。
- 计划列表、背包栏、地图、NPC档案、怪物档案、战斗面板、危机面板、NSFW 详细数值都属于事件驱动模块，未触发时沿用旧值。
- 未触发模块不需要专门写“无变化”占位；只有字段协议明确要求写 `无`、`0`、`未知` 时才写。

【统一触发定义】
- 下文若出现“剧情足够支持时，允许推断；一旦推断成立，必须写变量”，均默认指：对应模块已被本轮触发，且当前剧情、既有变量、世界书、场景逻辑已足够支持稳定判断时，才必须更新。
- “稳定判断”指能够高置信度推出当前最稳妥的结果，而不是为了填满面板进行过度脑补。
- 只要对应模块未被本轮触发，就默认沿用上轮值，不因为字段存在就强行改写。

【自定义开局兼容说明】
- 玩家可能只使用“同步角色”，然后自行发送开场白、开局问候语、开局动作或自定义场景。
- 这时必须把玩家输入视为正式剧情起点，并立刻据此更新变量。
- 不得因为没使用固定问候语模板，就让开局地点、天气、背包、关系事件长期留空。

变量更新规则:
  世界:
    年份:
      type: number
      range: 2030~
      check:
        - 原则上初始为2030，一般不频繁变化，只有出现明确跨年叙事时才更新

    世界基准时间:
      type: string
      check:
        - 作为世界观背景描述，原则上固定不变
        - 仅在整套世界观设定被重置或切换时更新

    单视角模式:
      type: boolean
      check:
        - 当前叙事严格保持单一角色第一人称时为 true
        - 明确切换到多角色并行叙事、群像视角或自由跳视角时改为 false
        - 若正文仍要求“角色A固定为{{user}}、角色B为当前主绑定对象/主要互动对象”的双角色主界面，则应优先保持 true
        - 未经玩家明确要求，不得因为旁支信息或他人行动自动切换为多视角

    允许NSFW:
      type: boolean
      check:
        - 仅在明确进入 NSFW 叙事并允许相关状态更新时改为 true
        - 普通暧昧、亲密试探、调情或身体接触不自动视为 true
        - NSFW 场景结束后应恢复为 false
        - 只有固定绑定 CP 且绑定对象在场时才允许改为 true
        - 非绑定对象、醉酒、暴走、药物、误会、强制引导等情况都不能绕过绑定限制开启 NSFW
        - 若当前剧情只到紧张气氛、擦边接触或情绪拉扯，仍保持 false

    高风险判定状态:
      type: boolean
      check:
        - 战斗、潜入、逃亡、危机爆发、暴走边缘、重大对峙时改为 true
        - 局势恢复稳定、脱离危险流程后改为 false
        - 高风险动作、调查、追踪、战斗、潜行、安抚、说服、异能控制等进入 D20 判定流程时优先改为 true
        - 只要当前轮仍存在未结算的危险后果，就不要过早恢复 false
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。
        - 即使正文未直写“进入高风险状态”，只要局势已经进入高风险动作、危险探索、未结算危机、持续战斗或高压对峙，就必须同步

    当前判定规则:
      type: string
      check:
        - 默认保持为 D20
        - 仅在明确切换判定系统时更新
        - 若大世界仍采用自动 D20 检定引擎，则不得随意改成其他文本

  世界信息:
    check:
      - 世界信息属于每轮必写核心模块，但只更新本轮真正发生变化的字段，不为整齐整组重写。
      - 时间 / 场所 / 天气 / 遭遇事件 / 当前场景标签 应围绕“当前核心场景状态”保持同步。
      - 若本轮只是轻微对话延续，没有形成新的场景、时间或遭遇变化，则允许部分字段沿用上轮值。

    时间:
      type: string
      format: YYYY-MM-DD HH:mm
      check:
        - 每当发生明确时间流逝时更新，包括等待、休息、赶路、战斗、治疗、长对话后
        - 短暂动作不需要机械性逐分更新，但重大场景切换后应体现时间变化
        - 不要跳过过长时间，除非叙事明确发生时间跃迁
        - 应与世界信息面板中的“第几天”保持一致
        - 若计划列表中存在明确约定时间，世界信息推进后应保证时序不自相矛盾
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。
        - 若根据等待时长、赶路过程、战斗时长、治疗跨度、长对话、场景跳转，能够高置信度推出时间已推进，即使正文没有逐字写出新的具体时分，也必须更新

    季节:
      type: string
      check:
        - 根据月份所在的季节进行变化
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。
        - 若时间推进或日期变化已经足以判断季节变化，即使正文未明说，也应同步更新

    场所:
      type: string
      check:
        - 角色进入新区域、建筑、房间、战区或关键地点时更新
        - 应写当前主要活动地点，而不是把经过的所有地点并列写入
        - 应与地图中的 current 节点一致
        - 若战斗面板地点已更新，世界信息的场所应同步
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。
        - 即使正文没有直接写出完整地点名，只要根据任务流程、势力驻地、人物所属、房间关系、当前路线、会面安排、战斗转移，能够高置信度推出当前最具体地点，也必须更新
        - 优先写当前最具体且最稳妥的地点；能确定房间/区域就写房间/区域，不能确定时再退到建筑或城市层级，不要因为不够完美就留空

    天气:
      type: string
      check:
        - 天候发生明显变化或场景切换到不同户外环境时更新
        - 室内场景可保持此前天气，除非叙事明确强调天气影响
        - 若世界信息面板仍采用“大气温 + 冷峻氛围描写”的写法，则字符串中应同时体现温度感与环境气氛

      type: string

        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。
        - 即使正文没有专门命名事件，只要当前局势已经形成明确的“遭遇类型”，例如盘查、警报、会面、跟踪、失控、感染征兆、援军抵达，也必须同步写入简短事件名

      type: number
      range: 1~

        - 仅在跨过自然日或叙事明确进入下一天时递增

        - 若时间字符串跨过日期，必须同步递增
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。

        - 若战斗面板中仍存在未结算敌方单位或回合数持续推进，则应保持 true
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。
        - 只要当前局势已进入持续交战、压制、追逐战、回合推进或未脱离危险接敌，就应保持 true；不要因正文短暂停顿就提前切回 false

    当前场景标签:
      type: string[]
      check:
        - 根据当前场景的性质、势力、任务状态、氛围添加关键词
        - 标签数量保持精简，一般 2~5 个即可
        - 场景核心改变时应同步删去过期标签
        - 应优先保留地点、所属势力、剧情阶段、危险等级四类标签
        - 若遭遇判定触发特殊事件，可临时加入事件标签，事件结束后移除
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。

      type: string

        - 若世界信息面板采用“遭遇判定：D20=[X] -> [触发/未触发] | [好事/坏事/中性]：[事件简称]”这类格式，则备注可用于补充未能写进主字段的后果
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。
        - 若已有无法归入其他字段但对当前局势重要的补充状态，也应同步备注，不要长期留空

      type: string

        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。

      type: string

        - 只要当前检定已能明确归属某属性，就必须同步，不要等全部结算后才补

    最近骰点:
      type: number
      range: 0~20
      check:

        - 无检定时为 0

      type: number
      check:

      type: string

        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。
        - 只要检定结果已能分级，就必须同步，不要只写数字不写等级

      type: string
      check:

        - 无额外后果时写无
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。
        - 只要本次判定已带来额外代价、收益或状态变化，就必须写入，不要只留在正文

    是否已结算:
      type: boolean
      check:
        - 新掷骰但后果尚未写入正文和面板时为 false
        - 一旦结果已经体现在正文与变量中，改为 true

  通用状态栏:

  危机面板:
    type: |-
      {
        [危机ID: string]: {
          类型: '感染'|'暴走'|'重伤'|'中毒'|'精神崩溃';
          对象: string;
          数值: string;
          危机值: number;
          阶段: string;
          阶段数据: { 当前: number; 总: number; };
          症状: string;
          处置: string;
          倒计时: string;
          倒计时数: number;
          倒计时单位: '无'|'回合'|'小时'|'天';
          状态: '进行中'|'缓解'|'稳定'|'解除'|'恶化';
          备注: string;
        }
      }
    check:
      - 出现真实危机时新建唯一危机ID，不要把多个危机混写在同一项里
      - 同一对象可以同时存在不同类型危机，但每个危机应单独编号
      - 初始占位危机仅在确无危机时保留，一旦出现真实危机应删除或不再使用占位项
      - 阶段数据是主字段，阶段字符串应同步显示为 当前/总
      - 倒计时数为 0 且状态为解除时，可移除该危机项或保留作记录
      - 危机变化应与通用状态栏、战斗面板中的伤势/暴走/污染状态互相呼应
      - 只有感染克锐特病毒、哨兵暴走、重伤（HP≤20%）、中毒、精神崩溃的人才应进入危机面板
      - 进入危机后，通用状态栏只保留结果值，不再重复阶段描述
      - 处置必须结合当前剧情资源，如引导安抚、绿核碎片、急救、抑制剂、撤离
      - 引导安抚需要进行 GDE 检定
      - 韩佳温是目前唯一可逆转高阶段感染的特殊例外
      - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。
      - 即使正文没有专门喊出“现在进入危机面板”，只要伤势、感染、暴走、中毒、精神崩溃已明确成立，就必须立即建危机项，不得只写正文不写变量

    危机面板.类型:
      type: "'感染'|'暴走'|'重伤'|'中毒'|'精神崩溃'"
      check:
        - 必须只使用这五类
        - 不要用自由文本替代类型值

    危机面板.对象:
      type: string
      check:
        - 必须写明危机对象
        - 若对象身份不明，感染默认按普通人感染处理，直到剧情明确其为觉醒者（异能者/引导者）
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。

    危机面板.数值:
      type: string
      check:
        - 暴走类优先写“暴走值XX/100”
        - 中毒类优先写“中毒层数XX层”
        - 其他类型可写无，也可写与危机相关的关键数值提示
        - 若大世界输出格式要求该行在部分类型可省略，变量层仍建议保留字符串占位，避免结构不稳定
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。

    危机面板.危机值:
      type: number
      range: 0~
      check:
        - 表示当前危机强度或推进程度
        - 危机加重时上升，缓解时下降
        - 不必所有危机都依赖同一数值尺度，但同一危机内部应保持自洽
        - 对暴走危机可直接与暴走值同步
        - 对感染危机可作为阶段推进强度辅助值使用
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。

    危机面板.阶段数据:
      type: "{ 当前: number; 总: number; }"
      check:
        - 当前必须在 1~总 之间
        - 危机推进时增加当前阶段
        - 缓解不一定直接倒退阶段，需依据叙事合理处理
        - 感染通常为 3 阶段
        - 暴走通常为 4 阶段，超过上限可进入 5/4 的转化结果
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。

    危机面板.阶段:
      type: string
      format: 当前/总 或 当前/总 · 阶段描述
      check:

        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。

    危机面板.症状:
      type: string
      check:
        - 必须写当前具体症状，不要只写抽象标签
        - 普通人感染 1/3：低烧、幻听、寒战、短暂眩晕
        - 普通人感染 2/3：绿色孢子纹路浮现、自主神经紊乱、情绪失控
        - 普通人感染 3/3：完全失智、身体扭曲异变、转化为低阶克锐特
        - 觉醒者（异能者）感染 1/3：轻度暴走反应、情绪波动、感知失衡、异能输出不稳
        - 觉醒者（异能者）感染 2/3：重度暴走反应、异能失控、敌我识别下降、攻击性增强
        - 觉醒者（异能者）感染 3/3：精神与肉体同时崩坏，扭曲成克锐特或死亡
        - 觉醒者（引导者）感染 1/3：低烧、幻听、精神疲惫、链接不稳
        - 觉醒者（引导者）感染 2/3：绿色孢子纹路浮现，精神干涉能力紊乱，引导精度下降
        - 觉醒者（引导者）感染 3/3：精神系统彻底崩坏，扭曲成克锐特或死亡
        - 未写明专属暴走反应的角色，使用通用暴走阶段症状
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。

    危机面板.处置:
      type: string
      check:

        - 处置要和当前资源、人物在场情况、背包内容一致
        - 不允许凭空写出当前不存在的治疗手段
        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。

    危机面板.倒计时数:
      type: number
      range: 0~
      check:
        - 有明确时限推进的危机才使用
        - 时间推进后递减
        - 归零时应触发恶化、爆发、结算或解除，不要归零后长期不处理
        - 感染相关危机默认围绕 24/48/72 小时黄金窗口推进

        - 剧情足够支持时，允许推断；一旦推断成立，必须写变量。

    危机面板.暴走规则:
      type: special
      check:
